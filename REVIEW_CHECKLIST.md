## 四、实施顺序合理性复核

### 4.1 依赖关系图

```
Phase 1 (技术债务清理)
    │
    ├─ 1.1 备份分支 (无依赖)
    ├─ 1.2 验证原始类 (无依赖)
    ├─ 1.3 检查引用 (依赖1.2)
    ├─ 1.4 删除文件 (依赖1.3)
    ├─ 1.5 编译验证 (依赖1.4)
    ├─ 1.6 修复PMD (无依赖)
    └─ 1.7 提交 (依赖1.5,1.6)
           │
           ▼
Phase 2 (代码质量)
    │
    ├─ 2.1 Spotless (依赖Phase 1)
    ├─ 2.2 Checkstyle (依赖Phase 1)
    ├─ 2.3 PMD (依赖1.6)
    ├─ 2.4 SpotBugs (依赖Phase 1)
    └─ 2.6 提交 (依赖2.1-2.4)
           │
           ▼
Phase 3 (测试覆盖)
    │
    ├─ 3.1 基线报告 (依赖Phase 2)
    ├─ 3.2 编写测试 (依赖3.1)
    ├─ 3.3 批量创建 (依赖3.2)
    ├─ 3.4 运行验证 (依赖3.3)
    └─ 3.5 提交 (依赖3.4)
           │
           ▼
最终验证 (依赖所有Phase)
```

### 4.2 依赖合理性评估

| 依赖关系 | 评估 | 说明 |
|----------|------|------|
| Phase 1 → Phase 2 | ✅ 合理 | 需先清理债务再提升质量 |
| Phase 2 → Phase 3 | ✅ 合理 | 需保证代码质量再写测试 |
| 1.2 → 1.3 → 1.4 | ✅ 合理 | 验证→检查→删除的安全流程 |
| 1.4 → 1.5 | ✅ 合理 | 删除后立即验证 |

**建议**: Phase 1.6 (PMD修复) 可与 Phase 2并行，无严格依赖。
